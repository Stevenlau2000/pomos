## 1. 项目信息

| 项 | 内容 |
|---|---|
| Language | 简体中文（与需求一致） |
| Programming Language | Next.js 14 + TypeScript + Tailwind CSS + ECharts + KaTeX（静态导出部署 GitHub Pages，basePath `/pomos`）；后端 FastAPI 仅本地 `localhost:8000`，**线上无法部署后端，必须离线降级** |
| Project Name | `pomos_chat_explain_enhancement` |
| 原始需求复述 | 用户在线上使用「问导师」时反馈：导师回答太简单——缺乏详细讲解、没有图表、没有动画、没有 step-by-step 步骤。要求：①详细文字讲解；②物理示意图；③原理动画（可播放的物理过程演示）；④一步一步说明。把问导师回答从「简略文本」升级为「结构化详细讲解（文字+图表+动画+step-by-step）」。 |

### 1.1 已拍板决策（PRD 必须遵守，不得推翻）

1. **讲解形态 = 静态图表 + 原理动画**：用 SVG/Mermaid 画力分析图/电路图/光路图/v-t 图等静态示意图 + KaTeX 公式，**并额外支持可播放的物理过程动画**（波动传播、抛体轨迹、电荷在电场中运动等）。最高直观度，架构复杂度最高。
2. **生成来源 = 云端 LLM 优先 + 离线降级（沿用既定决策）**：联网时腾讯混元生成结构化详细讲解（含图表/动画描述标记），断网降级到本地物理模板。

### 1.2 与既有增强 PRD 的关系（边界澄清）

- 本 PRD **建立在** `prd-enhancement-2026-07-19` 之上：复用其 REQ-CHAT-01 的「四模块讲义」结构（概念辨析/数理推导/图像分析/逻辑贯通）与 REQ-KB-02 的教材库。
- 不替代讲义模块：REQ-CHAT-01 是「生成讲义：xxx」独立意图；本 PRD 是把**同一套结构化讲解能力下沉到普通对话回答**（问导师的每一次答疑）。
- 本次新增增量 = 「讲解呈现质量」：结构化骨架、**图表渲染**、**原理动画**、云端结构化输出约定、教材联动标注。

## 2. 产品目标

**核心目标**：把「问导师」的回答从「一句话/几个要点」升级为「结构化、可解释、有图有动画的详细讲解」，显著提升物理直觉的建立效率与可理解性。

### 2.1 目标分解（正交、可度量）

| 编号 | 目标 | 说明 |
|---|---|---|
| G1 | 讲解结构化 | 每次解答遵循固定 step-by-step 骨架（六阶段），不再零散成段 |
| G2 | 内容详细度 | 文字讲解充分展开，含概念、推导链、图像描述、易错提示，而非结论堆砌 |
| G3 | 多模态呈现 | 至少覆盖静态图表（力分析/电路/光路/v-t 等）+ 可播放原理动画 |
| G4 | 联网/离线一致体验 | 云端质量高（需联网），离线稳定保底（纯本地模板），二者输出同一结构 |

### 2.2 成功指标（首版验收口径）

| 指标 | 云端 LLM（联网）目标 | 离线模板（断网）目标 |
|---|---|---|
| 解答含 ≥3 个结构化模块的比例 | ≥ 90% | ≥ 70% |
| 解答含 ≥1 图表或动画的比例 | ≥ 80% | ≥ 50% |
| step-by-step 六阶段骨架完整率 | ≥ 85% | ≥ 60% |
| 解答平均字数 | ≥ 300 字 | ≥ 150 字 |
| 教材引用标注率（P1 生效后可计） | ≥ 60% | ≥ 40% |
| 离线降级触发后图表不缺失率 | — | ≥ 80% |

> 度量方式：在 `MessageBubble` 渲染后采集结构化 `explain` 字段的模块/图表/公式数量，按会话抽样统计。

## 3. 用户故事（学生视角）

> 覆盖：联网/离线、力学/电磁/光学/热学/近代物理；每条标注「云端/离线」与「涉及需求」。

- **US-1（力学·联网）**：作为备考力学的高中生，我问「斜抛运动为什么水平方向匀速」，希望导师先拆解坐标系与受力、再列运动方程、画出力分析 SVG 与轨迹/速度-时间图、最后用一句话点出易错点「忽略空气阻力」，这样我能建立完整物理图像，而不是只看到结论公式。
- **US-2（电磁·联网）**：作为学电磁学的学生，我问「带电粒子在匀强磁场中偏转半径怎么求」，希望看到洛伦兹力推导（KaTeX 公式）、圆轨道静态 SVG、以及粒子做圆周运动的**可播放动画**，并标注引用的教材章节，帮我理解而非死记公式。
- **US-3（光学·离线）**：当我在无网络环境（GitHub Pages 静态站点）问「折射定律」，即使云端不可用，本地模板也应产出折射光路 SVG + 斯涅尔定律公式 + 易错点（如「入射角从空气量起」），保证离线也有图、可理解。
- **US-4（热学·联网）**：作为学热力学的学员，我问「卡诺循环效率为什么有上限」，希望看到四个过程的推导、p-V 图（静态或动画循环演示）、以及结论「实际热机 η<卡诺 η」，理解效率天花板来源。
- **US-5（近代物理·联网+追问）**：我问「时间膨胀怎么推导」，导师给出洛伦兹变换推导、时空图与教材引用；我进一步追问「这一步为什么」（P2），导师能针对某一步单独展开，不重写全文。

## 4. 需求池（P0 / P1 / P2）

> 优先级：P0=首版必须交付；P1=重要；P2=增强。
> 标注职责：**【云端】**=联网 LLM 生成结构化内容；**【前端】**=渲染图表/动画；**【离线】**=本地物理模板保底。

### P0（必须，首版交付）

- **P0-1 讲解结构化骨架**【前端+云端/离线】
  定义固定六阶段骨架：`问题拆解 → 概念辨析 → 数理推导 → 图像分析 → 结论 → 易错点`。云端 LLM 与离线模板均须按此骨架产出（模块可空但顺序固定），前端按阶段分段/折叠渲染。
- **P0-2 文字详细度提升**【云端+离线】
  - 云端：prompt 引导混元输出「展开式」讲解（每步含 reasoning，禁用「显然」「易得」式省略），输出受限在 `lib/llm.ts` 的 `maxTokens`（当前默认 1200，需评估上调至 ≥2000 以承载详细内容）。
  - 离线：`offlineTutor` / 新增离线讲解器需将 `OFFLINE_KB` 的要点扩写为含推导链的多句文本，而非仅 bullet。
- **P0-3 静态图表渲染**【前端，云端/离线提供图元数据】
  - 复用既有 KaTeX（`InlineMath`/`BlockMath`）渲染公式。
  - 新增**物理专有 SVG 组件库**（力分析图、光路图、场线图、轨迹图、电路简图），由前端按图类型选择器渲染；通用图（如 p-V 曲线、流程图）可复用既有 Mermaid（` ```mermaid ` 通道已存在）。
  - 首版至少覆盖高频题型图示：力学受力分析、抛体/运动轨迹、电路、折射/反射光路、v-t/p-V 图。
- **P0-4 云端 LLM 结构化输出约定**【云端+前端】
  - 混元须返回**带图表/动画元数据标记的结构化 JSON**（非纯文本），前端解析后渲染富卡片，并保留 markdown 纯文本兜底。
  - 建议 schema 版本 `PomosExplainV1`（详见 §5.1）。讲解意图走非流式 `chatCompletion` 取 JSON，再由前端逐步揭示（模拟 step-by-step），避免增量解析半截 JSON。
  - 解析失败时降级为「markdown 纯文本渲染」（沿用 `MessageBubble` 现有通道）。

### P1（重要）

- **P1-1 可播放原理动画**【前端+云端/离线】
  - 定义**动画描述 DSL / 参数 schema**（如 `{type:"projectile", params:{v0,theta,g}}`、`{type:"wave", params:{lambda,f}}`），前端用 SVG + JS/CSS（requestAnimationFrame）渲染简单物理过程：匀速/匀加速运动、波动传播、折射、电荷在电场中偏转等。
  - 云端 LLM 生成该 DSL 参数；离线模板预置少量典型动画参数。
- **P1-2 离线降级也尽量产出图表**【离线+前端】
  - 离线讲解器预置典型图（受力/光路/v-t 等）的 SVG 参数，断网时仍渲染静态图；动画在离线优先渲染**静态首帧 + 播放按钮**（保底不黑屏）。
- **P1-3 参考教材联动**【云端/离线+前端】
  - 讲解中标注引用的教材章节（如「赵凯华《新概念物理教程·力学》§3.2」），复用 `lib/textbookRetriever.ts` 的 `searchTextbooks` 返回出处；前端在「教材引用区」以可点击标签呈现。

### P2（增强）

- **P2-1 动画交互**：参数可拖拽实时重算（滑块调节 v0/θ 等，动画即时响应）。
- **P2-2 多语言**：讲解文本支持中英切换（沿用 `lib/i18n` 与 `coach_language` 机制）。
- **P2-3 追问「这一步为什么」**：学生对某一步发起追问，导师针对该 step 单独展开，不重输全文（需 step 可寻址 id）。

### 5.3 渲染要点

- **标题区**：问题标题 + 模式徽标（云端/离线），让学生感知来源与可信度。
- **step-by-step 区**：六阶段按序分段；默认首屏展开前 2 步，其余折叠（点击展开），前端以 150ms 间隔逐步揭示模拟「step-by-step」节奏。
- **图表区**：`diagram.kind` 映射到 SVG 组件（force/circuit/light/vt/trajectory）或 Mermaid 渲染器（mermaid）。
- **动画区**：`animation` 非空时渲染 `AnimationPlayer`（SVG+JS/CSS），含 播放/暂停/重播 控件；离线动画保底渲染首帧 + 播放按钮。
- **公式区**：内联 `InlineMath` / 块级 `BlockMath`（KaTeX），与现有 `RichText` 一致。
- **易错点高亮**：`misconception` 非空时以 `destructive` 底色卡片呈现，呼应「降低踩坑率」目标（G4 同既有 HPCAS 评估区风格）。
- **教材引用区**：`source_refs`/`sources` 以可点击标签呈现，复用 `textbookRetriever` 出处。

## 6. 待确认问题（PM 需澄清，不推翻两条拍板决策）

1. **动画 DSL 由谁生成？** 建议：云端 LLM 生成 `animation` 参数（质量高、灵活），离线用预置模板参数（稳定）。需确认 LLM 生成的 DSL 是否需经前端 schema 校验/沙箱。
2. **离线降级动画的保底策略？** P1-2 要求离线也出图；动画在离线是否仅渲染静态首帧+播放按钮，还是允许纯 CSS 动画（无需 LLM）？需明确离线动画的「最低可用」定义。
3. **图表库选型：Mermaid vs 手绘 SVG？** 建议分层：力分析/光路/场线/轨迹等物理专有图用手绘 SVG 组件库（Mermaid 不擅长）；电路流程/KVL、p-V/v-t 可用 Mermaid（已集成）。需架构师确认 SVG 组件库的维护成本与 bundle 体积（静态导出约束）。
4. **云端 JSON 与流式体验如何兼顾？** 建议讲解意图走非流式 `chatCompletion` 取 JSON + 前端逐步揭示。是否接受「先整段返回再逐步展开」替代「逐字流式」？
5. **讲解意图识别如何与 REQ-CHAT-01 区分？** 现有「生成讲义：xxx」走 `lecture.ts` 四模块；本次普通问答的「详细讲解」是否复用同一四模块骨架但走对话通道（非独立讲义）？需避免两套渲染逻辑分裂。
6. **`ChatMessage` 结构扩展影响面？** 现 `content: string` 需扩展为携带 `explain?: PomosExplainV1`，涉及 `api.ts`/`offlineApi.ts` 返回类型、`MessageBubble`、历史存储（IndexedDB）序列化，需确认是否影响既有历史消息兼容（旧消息无 `explain` 应走原 markdown）。
7. **bundle / 静态导出体积约束？** SVG 动画组件库 + Mermaid 已较大，新增物理图组件与动画引擎是否触发 GitHub Pages 体积告警？需评估按需加载（dynamic import）策略。
8. **`maxTokens` 是否上调？** 详细结构化讲解需更长输出，当前 `lib/llm.ts` 默认 1200，建议讲解意图上调至 ≥2000，需确认混元配额/成本。

## 7. 职责边界速查表（落地指引）

| 能力 | 云端 LLM（联网） | 前端渲染 | 离线模板（纯本地） |
|---|---|---|---|
| 结构化骨架 | 按六阶段生成 steps | 分段/折叠渲染 | 按六阶段组装 steps |
| 详细文字 | prompt 引导展开推导 | 文本渲染 | 扩写 OFFLINE_KB 要点 |
| 公式 | 输出 KaTeX | KaTeX 渲染 | 预置公式 |
| 静态图表 | 输出 diagram spec | SVG/Mermaid 渲染 | 预置 diagram spec |
| 原理动画 | 生成 animation DSL | SVG+JS/CSS 播放 | 预置动画参数/首帧 |
| 教材引用 | 生成 source_refs | 可点击标签 | 调 `textbookRetriever` |

> 降级链路（沿用 `lib/llm.ts` 约定）：`!isConfigured() / navigator.onLine===false / fetch 非2xx / 超时(8s) / 抛错` → 回退离线讲解器，输出同构 `PomosExplainV1`（`mode:"offline"`）。
